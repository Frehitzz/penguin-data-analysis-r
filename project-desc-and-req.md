Grading Systems
1. Code Quality &amp; Structure (20%)
 Readability: Code is clean, indented properly, and easy to read.
 Commenting: The code includes meaningful comments explaining why steps are taken, not
just what the code does.
 Best Practices: Uses modern R syntax (e.g., tidyverse over base R where appropriate),
adheres to a consistent style guide (like the tidyverse style guide), and avoids hardcoding
values.
 Modularity: Uses functions to avoid repetition (DRY - Don&#39;t Repeat Yourself principle).
2. Functional Correctness (40%)
 Reproducibility: The code runs from start to finish without errors, with no dependencies on
local file paths (setwd() should be avoided).
 Data Handling: Data is loaded, cleaned, and tidied efficiently. Missing values and outliers are
handled correctly.
 Analysis/Modeling: Appropriate statistical methods or models are chosen for the data. The
models are correctly implemented and evaluated.
 Debugging: Code handles edge cases and does not produce nonsensical results.
3. Data Visualization &amp; Reporting (30%)
 Visualizations: Plots are labeled appropriately (axes, titles, legends), colors are used
effectively, and graphs are optimized for interpretation (e.g., using ggplot2).
 Narrative/Interpretation: The analysis explains the &quot;so what?&quot;—interpreting the statistical
findings in the context of the research question.
 Report Quality: The final output (e.g., R Markdown, Quarto, Shiny App) is well-organized,
polished, and free of unnecessary code chunks or warnings. [1]
4. Workflow &amp; Documentation (10%)
* Project Structure: Files are organized logically (e.g., folders for data/, scripts/, output/).
* README: A README.md file is present, describing what the project does and how to run it.
